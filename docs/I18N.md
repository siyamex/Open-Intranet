# Internationalization — extraction progress

OpenIntranet's i18n layer (`App\Core\Lang`, `lang/{code}.php`, the `__()`
helper) is fully wired: per-user language picker in the avatar menu, an
admin "Languages" settings tab to enable/disable locales and set the
default, `dir="rtl"`/`dir="ltr"` on `<html>` driven by each language's
`is_rtl` flag, logical CSS properties (`inset-inline-*`, `margin-inline-*`)
so the shell mirrors automatically, and `IntlDateFormatter`-based date
formatting with a fallback.

Three locales ship: **en** (reference), **dv** (Dhivehi, RTL),
**ar** (Arabic, RTL) — dv/ar are machine-assisted drafts pending native
review.

## What's translated now

The `__()` helper and `lang/*.php` files cover the **app shell**: sidebar
navigation labels, navbar search placeholder, avatar menu items, sign-in
page, and dashboard section headings. This is enough to prove the plumbing
end-to-end — switch languages from the avatar menu and the whole shell
follows, RTL layout included.

## What's not yet extracted

The bulk of view text (~120 files) is still hard-coded English strings, by
design: the pack's own instructions call for this to be done "file by file"
with a tracked progress table, which is what this file is. Extracting all of
it in one pass would mean touching every view in the app; instead each
module's strings should be pulled out the next time that module is touched,
using the pattern already established in the shell:

| Area | Status |
|---|---|
| App shell (nav, navbar, avatar menu) | ✅ Done |
| Auth pages (login, password reset) | 🟡 Partial (login labels only) |
| Dashboard widgets | 🟡 Partial (headings only) |
| News | ⬜ Not started |
| Documents | ⬜ Not started |
| Directory / org chart | ⬜ Not started |
| Events / polls / kudos / wiki / requests | ⬜ Not started |
| Admin panel | ⬜ Not started (admin is expected to read English) |

## Adding a string

1. Add the key to `lang/en.php` (the reference — every other file must have
   the same keys).
2. Add the same key to `lang/dv.php` and `lang/ar.php` (or leave it out —
   `Lang::get()` falls back to English for missing keys, so nothing breaks).
3. Use it in the view: `<?= e(__('your.key')) ?>` or with params:
   `__('dashboard.welcome', ['name' => $name])` (`:name` placeholders).

## Adding a language

Insert a row into the `languages` table (`code`, `name`, `native_name`,
`is_rtl`), add `lang/{code}.php` with at least the keys used by the shell,
then enable it under **Admin → Settings → Languages**.

## Content translation (news)

A `news_translations` table (migration `020_i18n.sql`) keys translated
title/excerpt/body by `(news_id, locale)` for a future per-post language-tab
editor UI — the schema is in place; the editor tabs are not yet built.
